# User-Facing Response Contract

Read this before any user-facing orchestrator reply. Follow it verbatim.

Не импровизируй новый layout. Повторяй один и тот же markdown-шаблон в каждом user-facing ответе.

## Fixed Section Order

1. level-1 heading verdict: `# ACCEPTED`, `# REJECTED`, `# BLOCKED` или `# NEEDS_FOLLOWUP_TASK`;
2. quoted line `> **Human touch:** Yes` или `> **Human touch:** No`;
3. если нужен ответ пользователя, блок `**Вопросы к пользователю**`;
4. если нужен user dispatch, блок `**Действия для пользователя**`;
5. task handoff blocks;
6. `## Роудмеп`;
7. `## Порядок постановки`;
8. всегда самым последним `## Простыми словами`.

## Hard Rules

- после verdict line не вставляй длинный explanatory paragraph;
- `Human touch` не означает обычную постановку задач пользователем;
- если от пользователя требуется только открыть новый worker thread, продолжить существующий thread, вставить task spec или запустить обычный agent run, ставь `> **Human touch:** No`;
- `> **Human touch:** Yes` ставь только если дальше нужен реальный human-only шаг: содержательное решение, ручной ввод секрета или credential, действие во внешнем UI, ручная выборка/разметка данных, approval/rejection с бизнес-контекстом или другой шаг, который нельзя свести к обычному dispatch;
- если в ответе есть блок `**Вопросы к пользователю**`, `> **Human touch:** Yes` обязателен;
- если product-visible behavior, priority, scope boundary или success criteria не определены в tracked docs и их рискованно домысливать, подними вопрос пользователю вместо invent-by-default;
- если implementation uncertainty можно сначала закрыть bounded research-проходом по repo/docs/official sources, не показывай `**Вопросы к пользователю**` в этом ходе;
- в `**Действия для пользователя**` пиши только короткие механические действия пользователя. Не дублируй там строку `Human touch` из начала ответа;
- пользовательский блок не должен требовать от пользователя понимать task scopes, file ownership, lane separation, safe parallelism, routing policy или write conflicts;
- не пиши там `убедись`, `не смешивай`, `держи отдельно`, `проверь`, `следи чтобы`, `параллель только если` и любые аналогичные инструкции, которые перекладывают orchestration responsibility на пользователя;
- если для safe parallelism нужны carve-outs, thread separation или write boundaries, зафиксируй это в task specs, `Scope`, `Не трогать`, thread routing и dispatch order, а не в инструкциях пользователю;
- наличие блока `**Действия для пользователя**` само по себе не делает `Human touch` равным `Yes`;
- не задавай open-ended вопросы вроде `как лучше`, `что думаешь`, `что ты хочешь видеть` без предварительно подготовленных вариантов;
- каждый вопрос в `**Вопросы к пользователю**` должен быть bounded decision с явными вариантами ответа;
- первым вариантом всегда ставь рекомендуемый вариант и явно помечай его как `Рекомендуется`;
- один вопрос должен решать одну блокирующую развилку; не склеивай несколько независимых продуктовых решений в одну расплывчатую формулировку;
- если вопрос можно снять из tracked docs, roadmap, product brief или nearby source, не задавай его пользователю;
- если вопрос можно снять через отдельную research-task и это дешевле, чем отвлекать пользователя, сначала ставь research-task;
- explanatory prose держи только в самом конце, в `## Простыми словами`;
- перед task handoff blocks не вставляй user-facing prose с lane analysis, file grouping, warnings или operational caveats, если эти ограничения должен держать сам orchestrator;
- fenced blocks разрешены только для task specs;
- после task spec никогда не заворачивай roadmap, progress, acceptance, summary или любой другой текст в fenced block;
- после закрытия последнего task spec не открывай больше ни одного fenced block в этом сообщении;
- `## Порядок постановки` всегда пиши обычным markdown numbered list, а не code block;
- если после task spec у тебя получается ещё один блок с language label вроде `text`, это сломанный ответ; перепиши сообщение до отправки;
- перед отправкой делай self-check: количество fenced blocks в ответе должно быть ровно равно количеству task specs;
- `## Роудмеп` обязателен в каждом сообщении, даже если downstream task сейчас нет;
- в `## Роудмеп` показывай реальные фазы из роудмепа в порядке `R1`, `R2`, `R3` и далее;
- `## Роудмеп` должен отражать полный scope проекта до MVP, а не только текущую dispatch wave;
- у каждой фазы пиши heading вида `### Rn - status`;
- сразу под heading каждой фазы пиши одну короткую строку курсивом с названием или смыслом фазы;
- если status фазы terminal, например `done`, `skip`, `backlog`, `cancelled`, не расписывай её задачи;
- если status фазы не terminal, расписывай задачи этой фазы сразу под ней;
- если у нетерминальной или planned фазы задач нет, не скрывай это новой локальной таской; сначала исправь сам роудмеп;
- не используй отдельный блок `Текущая фаза`;
- не заменяй полный фазовый роудмеп коротким пересказом только ближайшей фазы;
- не делай отдельный блок `Итого по прогрессу`;
- `## Порядок постановки` должен уже включать и факт параллельности, и ближайшие задачи. Не делай отдельные блоки `Следующая задача в постановку сейчас` или `Параллельность`;
- первый пункт внутри `## Порядок постановки` обязан быть только `1. Параллельность: Да` или `1. Параллельность: Нет`;
- следующие пункты внутри `## Порядок постановки` должны содержать только task codes без объяснений;
- если это follow-up после reject, пиши один пункт со строкой follow-up, где оба task code оформлены через inline code;
- если задачи ставятся параллельно в одной волне, пиши их в одной строке через `+`, а оба task code оформляй через inline code;
- любой task code вне fenced block оборачивай в inline code, чтобы он рендерился серой плашкой;

## `Вопросы к пользователю` Format

- heading блока всегда literal `**Вопросы к пользователю**`;
- каждый вопрос нумеруй обычным markdown numbered list;
- сразу после строки вопроса перечисляй варианты ответов как `A.`, `B.`, `C.` и далее;
- у первого рекомендуемого варианта добавляй суффикс `(Рекомендуется)`;
- формулируй варианты так, чтобы пользователь мог просто ответить буквой или скопировать формулировку варианта без дополнительного сочинения;
- если вариантов много, перечисляй столько, сколько реально нужно; не ужимай вопрос до `A/B`, если в продукте реально `5+` рабочих политик;
- не оставляй вопрос без вариантов вообще.

## Research-First Rule

- `**Вопросы к пользователю**` появляются только после bounded research-pass или отдельной research-task по этой неопределённости;
- если research-task уже выдана, не задавай пользователю тот же вопрос в том же сообщении, если только не нужен параллельный human-only выбор;
- пользователь не должен подменять собой docs/API/code research.

## `Простыми словами` Rules

- `## Простыми словами` обязателен.
- Пиши этот блок как короткий статус-апдейт для нетехнического product manager.
- Объясняй только: что решили по задаче, что это дало проекту или MVP, что ещё мешает, какой следующий шаг.
- Не тащи сюда file paths, module names, class names, function names, test names, commands, commit hashes, branch/worktree state, env vars, flags, runtime internals и другой code-level жаргон.
- Если в `## Простыми словами` полезли технические доказательства, команды или внутренние implementation details, убери их наверх в технические секции и перепиши блок заново.
- Если задача accepted, простым языком напиши, что это была за задача и какой прогресс по проекту она дала.
- Если задача rejected или blocked, простым языком напиши, что именно не доказано и что это значит для общего прогресса.
